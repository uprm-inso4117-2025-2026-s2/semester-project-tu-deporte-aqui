A Methodical Testing Strategy follows a predefined quality standard — checklists or a fixed set of test conditions — to verify that all database attributes and frontend integrations are correctly implemented. This document applies that strategy to the Tu Deporte Aquí project, where the database schema, FK relationships, seed data, RLS policies, and frontend utility functions provide a well-defined test basis that maps naturally to checklist-based verification in support of the project's Brand Confidence and Risk Reduction objectives.

=== Database Schema Reference

The following schema was extracted from the live Supabase project and constitutes the test basis for all checklist items. The `teams` table holds `id` (uuid, NOT NULL), `name` (text, NOT NULL), `league` (text, NOT NULL), `city` (text, nullable), and `logo_url` (text, nullable). The `athletes` table holds `id` (uuid, NOT NULL), `team_id` (uuid, nullable), `first_name` (text, NOT NULL), `last_name` (text, NOT NULL), `position` (text, nullable), and `number` (integer, nullable). The `games` table holds `id` (uuid, NOT NULL), `home_team_id` (uuid, nullable), `away_team_id` (uuid, nullable), `league` (text, nullable), `season` (text, nullable), `game_date` (date, nullable), `start_time` (time, nullable), `venue` (text, nullable), `status` (text, nullable), `home_score` (integer, nullable), and `away_score` (integer, nullable). The `standings` table holds `id` (uuid, NOT NULL), `team_id` (uuid, nullable), `league` (text, nullable), `season` (text, nullable), `wins` (integer, nullable), and `losses` (integer, nullable). The `news` table holds `id` (uuid, NOT NULL), `team_id` (uuid, nullable), `athlete_id` (uuid, nullable), `game_id` (uuid, nullable), `league` (text, nullable), `title` (text, NOT NULL), `content` (text, NOT NULL), `source` (text, nullable), `source_url` (text, nullable), and `published_at` (timestamptz, nullable). The `profiles` table holds `id` (uuid, NOT NULL), `display_name` (text, nullable), `avatar_url` (text, nullable), and `is_admin` (boolean, nullable). The `comments` table holds `id` (uuid, NOT NULL), `user_id` (uuid, nullable), `news_id` (uuid, nullable), `content` (text, NOT NULL), and `published_at` (timestamptz, nullable).

=== Row Level Security Policy Reference

Five tables — `teams`, `games`, `athletes`, `news`, and `standings` — carry public SELECT policies that allow any caller to read their rows without authentication. The `profiles` table carries a restrictive SELECT policy keyed to `auth.uid() = id`, meaning a user can only read their own row and cross-user reads are blocked by design. The `comments` table has no RLS policy defined. If RLS is enabled on the table without a policy, all reads are silently blocked, which would break any frontend component that displays comments. This must be confirmed and resolved before the checklist can be considered complete.

=== Frontend Test Results

All tests were executed directly against the project source in `tu-deporte-aqui/utils/`.

`scoreUtils.test.js` — 17 passed, 0 failed. `formatScore` correctly returns a formatted score string for valid inputs and `"Score unavailable"` when either score is null or undefined. `mapGameStatus` correctly maps `"FT"` and `"FINAL"` to `"Final"`, `"LIVE"` and `"IN_PROGRESS"` to `"Live"`, `"SCHED"` to `"Scheduled"`, and returns `"Unknown"` for unsupported or empty inputs. `isScoreComplete` returns `true` only when both scores are non-null and non-undefined. `isStaleTimestamp` correctly identifies timestamps outside the configured threshold, null values, and invalid strings as stale.

`scoreIntegration.test.js` — 7 passed, 0 failed. `buildDisplayGame` correctly assembles the display object for a live game with a complete score, a final game, a game with a missing home score, a game with a stale timestamp, a game with an unsupported status, a game with missing team names, and a game with an invalid timestamp string. In all cases the output fields matched the expected values exactly.

=== BDD Scenario Traceability

All nine scenarios in `tu-deporte-aqui/utils/.feature` trace to passing tests. The scenario for displaying a valid score for a live game is covered by I-01. Score unavailable for a missing home score is covered by U-02 and I-03. Score unavailable for a missing away score is covered by U-03. Status mapping from `"FT"` to `"Final"` is covered by U-04 and I-02. Status mapping from `"IN_PROGRESS"` to `"Live"` is covered by U-07. Status mapping from `"SCHED"` to `"Scheduled"` is covered by U-08. Status mapping from `"POSTPONED"` to `"Unknown"` is covered by U-09. Marking a game stale due to an old timestamp is covered by U-15 and I-04. Marking a game stale due to an invalid timestamp is covered by U-17 and I-07.
