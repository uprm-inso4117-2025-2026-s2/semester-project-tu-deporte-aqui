= Design Session-Based Exploratory Testing Charters for Database Inconsistencies

Author: Luis Sorrentini


An Exploratory Testing Charter strategy applies structured, session-based human-driven exploration to systematically uncover hidden data anomalies, orphaned connections, and broken API integrations directly within the Supabase dashboard. This document applies that strategy to the Tu Deporte Aquí project, where the database schema, FK relationships, nullable field risks, RLS policies, and Milestone 3 stability requirements provide a well-defined target space for structured defect discovery without altering the active database schema. The goal is to shift the team from reactive bug-fixing to proactive, timeboxed defect discovery aligned with the course lecture's session testing model, where the tester keeps a structured log to avoid unnecessary repetition of tests and surfaces findings that automated scripts typically miss.

=== Acceptance Criteria

This standalone document is saved in the research folder of the documentation branch and contains all required sections using correct AsciiDoc text formatting headers. Three formal session charters are defined in the Session Testing Charters section, each containing a mission statement, a clearly scoped target area, a 45-minute time limit, start conditions, exploration notes guidance, and a bugs and risks log structure. Charter 1 targets foreign key cascading failures on deleted or orphaned athletes. Charter 2 targets null-state handling in live game score fields. Charter 3 targets broken API integrations between the Supabase backend and frontend data-consuming components. Each charter follows the session-based format demonstrated in the lecture slides. The Defect Lifecycle Protocol section defines exactly how a discovered database anomaly transitions through the standard defect lifecycle taught in the course: New, Assigned, Fixed, Ready for Test, and Closed, with documented branches for Rejected and Failed Retest. Each stage identifies the responsible role and the specific action that triggers the transition, matched to the Milestone 3 team structure.

=== Testing Plan

All three charters were independently evaluated against the structural QA guidelines from the lecture. Each charter mission is phrased as a concrete exploration directive — "investigate", "probe", "verify" — rather than a vague goal, and each names specific tables, columns, or API routes as the target area, making the mission executable directly from the Supabase dashboard without additional setup. Charter 1 maps directly to Team 2's responsibility to resolve database inconsistencies and validate database relationships and data integrity. Charter 2 maps to Team 3's responsibility to fix bugs in `/games` and improve loading and error handling. Charter 3 maps to Team 2's responsibility to fix broken API integrations and ensure all frontend teams can reliably access data. No charter requires a major schema redesign, satisfying the Milestone 3 constraint. All three charters are scoped to 45-minute unbroken sessions, matching the session testing model from the lecture, and each scope is narrow enough to explore meaningfully within that window. The Defect Lifecycle Protocol and Defect Report Reference sections align with the six characteristics of a good defect report from the lecture: Objective, Specific, Concise, Reproducible, Explicit, and Persuasive. All three charters are written such that a team member who was not present during authoring can read the mission, open the Supabase dashboard, and execute a real testing session without additional briefing.

=== Session Testing Charters

The following three session charters are written in the structured format from the course lecture's session-based testing model. Each charter defines a mission, a target area, a time limit, start conditions, exploration guidance, and a log structure for bugs and risks. The tester keeps this log during the session to avoid unnecessary repetition of tests and to produce a reproducible record of what was covered.

Charter 1 is titled: Investigate Foreign Key Cascading Failures on Deleted or Orphaned Athletes. The mission is to explore the `athletes`, `news`, `comments`, and `games` tables in the Supabase dashboard to determine whether deleting or soft-removing an athlete record leaves orphaned FK references in related tables that would cause silent data anomalies or frontend crashes. The target areas are the `athletes` table (`id`, `team_id`), the `news` table (`athlete_id` FK), the `comments` table (`user_id` FK), and the `games` table (`home_team_id`, `away_team_id` FKs). The time limit is 45 minutes, unbroken. The start condition is that the tester has read access to the Supabase Table Editor and SQL Editor and has confirmed that at least one athlete record is linked to at least one news article via `athlete_id`. The exploration guidance is: navigate to the `athletes` table in the Table Editor and identify any rows where `team_id` is NULL — these are orphaned athletes already present in the seed data. Then navigate to the `news` table and filter rows where `athlete_id` is not NULL, noting how many articles reference athletes. Run a SQL query joining `news.athlete_id` to `athletes.id` using a LEFT JOIN and inspect any rows where the athlete side returns NULL — these represent broken FK references visible to the frontend. Repeat the process for `comments.user_id` joined to `profiles.id`. For each finding, note the table, column, row ID, and the frontend component that would consume that data. The bugs and risks log should record: bug ID, table affected, column, row UUID, observed anomaly, and the frontend component at risk. Open risks — questions that remain unresolved after the session — should be listed separately at the end of the log.

Charter 2 is titled: Probe Null-State Handling in Live Game Score Fields. The mission is to explore the `games` table in the Supabase dashboard and systematically identify all rows where `home_score`, `away_score`, `status`, `start_time`, or `game_date` are NULL, then trace each null field to the frontend component that renders it and assess whether a crash or silent failure would result. The target areas are the `games` table columns `home_score` (integer, nullable), `away_score` (integer, nullable), `status` (text, nullable), `start_time` (time, nullable), and `game_date` (date, nullable), and the `LiveGamesSection` component in `home/page.tsx`. The time limit is 45 minutes, unbroken. The start condition is that the tester has read access to the Supabase Table Editor and has opened `home/page.tsx` in a side window for reference. The exploration guidance is: navigate to the `games` table in the Table Editor and sort by `status`. Identify rows with `status = 'scheduled'` and confirm whether `home_score` and `away_score` are NULL for those rows — the seed data contains at least one such row. Then identify any rows where `status` itself is NULL and note how the `LiveGamesSection` badge label would behave with no status value. Sort by `game_date` and identify any rows where `game_date` or `start_time` is NULL and note what the time display would render. Run a SQL query selecting all games where at least one of the five target columns is NULL and count the results. For each NULL combination found, write one entry in the bugs log describing which column was null, what the frontend would render, and whether a crash or "Invalid Date" or "null" string would appear. The risks log should capture any NULL combinations not covered by the existing seed data that could appear in production.

Charter 3 is titled: Verify Broken API Integrations Between Supabase and Frontend Data-Consuming Components. The mission is to explore the Supabase dashboard's API documentation panel and RLS policy table to identify any mismatch between the public read policies applied to the five publicly readable tables and the data shapes that `home/page.tsx`, `/games`, `/news`, and `/standings` pages expect to receive, with a focus on finding silent permission failures or missing JOIN paths that would cause components to receive empty arrays instead of data. The target areas are the RLS policies on `teams`, `games`, `athletes`, `news`, `standings`, `profiles`, and `comments`, the Supabase API documentation panel for each table, and the frontend pages `home/page.tsx`, `app/games/page.tsx`, `app/news/page.tsx`, and `app/standings/page.tsx`. The time limit is 45 minutes, unbroken. The start condition is that the tester has access to the Supabase Authentication and Policies panel and has confirmed the five public SELECT policies are active. The exploration guidance is: open the Policies panel and confirm that `teams`, `games`, `athletes`, `news`, and `standings` each show an active public SELECT policy with condition `true`. Then navigate to the `comments` table policy panel and confirm whether any SELECT policy exists — prior research identified no policy on this table, which means reads may be silently blocked if RLS is enabled. For each of the five publicly readable tables, open the API documentation panel and inspect the auto-generated column list to verify it matches the confirmed schema. Run a raw SQL SELECT against each table without authentication context using the anonymous key simulation in the API panel and confirm that rows are returned. For `comments`, run the same test and note whether rows are returned or the result is empty due to a missing RLS policy. Log each finding as either a confirmed integration bug or an open risk. The bugs log should record: affected table, policy finding, expected frontend behavior, actual API response, and the team responsible per Milestone 3 assignments.

=== Defect Lifecycle Protocol

This section defines exactly how a discovered database anomaly transitions through the standard defect lifecycle from the course lecture. The lifecycle applies to any bug found during a charter session and must be followed before a defect can be considered resolved. The five primary stages are New, Assigned, Fixed, Ready for Test, and Closed. Two branch stages — Rejected and Failed Retest — handle cases where the defect cannot be reproduced or the fix does not hold.

The New stage is entered by the Tester at the moment a defect is logged during a charter session. The tester raises a new defect report using the format defined in the Defect Report Reference section below and assigns it the status New. The report must include the table name, column name, observed anomaly, steps to reproduce from the Supabase dashboard, expected behavior, and actual behavior. The defect is then submitted to the Development Lead for review.

The Assigned stage is entered by the Development Lead after reviewing the defect report and confirming the anomaly is reproducible and within scope for Milestone 3. The Development Lead assigns the defect to the appropriate developer based on team responsibilities: database inconsistencies and FK issues go to Team 2, game and standings rendering bugs go to Team 3, and news and profile rendering bugs go to Team 4. The status changes from New to Assigned. If the Development Lead cannot reproduce the anomaly using the steps provided, the defect transitions to Rejected instead.

The Rejected stage is a branch entered by the Development Lead when the defect cannot be reproduced. The report is returned to the Tester with notes explaining what was attempted. The Tester reviews the steps to reproduce and either closes the defect if the anomaly no longer appears, or rewrites the report with more explicit steps and resubmits it as a new New entry. If the Tester confirms the defect is no longer present, it transitions directly to Closed.

The Fixed stage is entered by the Developer after implementing a resolution for the assigned defect. The fix must not involve a major schema redesign per Milestone 3 constraints. The Developer marks the defect as Fixed and notifies the Release Manager that the fix is ready for the software build to be released to testing.

The Ready for Test stage is entered by the Tester after the fixed build is released to the testing environment. The Tester re-executes the original steps to reproduce from the defect report and checks whether the anomaly is still present. If the defect is confirmed fixed, it transitions to Closed. If the defect is still present, it transitions to Failed Retest.

The Failed Retest stage is a branch entered by the Tester when the fix did not resolve the anomaly. The Tester adds a retest note to the defect report describing what was observed during retesting and returns the defect to Assigned status. The Development Lead reassigns it to the Developer for a second fix attempt. This cycle repeats until the defect passes retest.

The Closed stage is the terminal state, entered by the Tester after confirming that the defect is no longer reproducible in the testing environment. The defect report is marked Closed and retained in the research folder as a record of the resolved anomaly.

=== Defect Report Reference

This section defines the standard format for a defect report produced during a charter session, based on the Elements of a Defect Report from the course lecture. Every report logged during a charter session must include all required fields. Optional fields should be included whenever the tester has sufficient information.

The Title field is required and must be unique, concise, and descriptive enough that the defect can be identified and discussed without reading the full report. A good title names the table, the column, and the observed symptom. For example: "`news.athlete_id` returns NULL JOIN result causing silent empty render in `NewsSection`" is a good title. "News broken" is not.

The Severity field is required and must be one of: Critical (frontend crash or data loss), High (silent wrong data displayed to user), Medium (degraded display such as a blank field or placeholder shown), or Low (cosmetic or edge-case only). Severity is assessed based on impact to the end user, not to the developer.

The Status field is required and must reflect the current lifecycle stage: New, Assigned, Fixed, Ready for Test, Failed Retest, Rejected, or Closed.

The Initial Configuration field is required and must describe the exact state of the Supabase dashboard and the relevant table before the steps to reproduce are followed. This includes which table was open, what filter or sort was applied, and the row ID of the affected record.

The Software Configuration field is required and must identify the Supabase project name, the branch being tested, and whether the tester was using the anonymous key or an authenticated session during exploration.

The Steps to Reproduce field is required and must be an ordered series of explicit steps that any team member can follow from the Supabase dashboard to observe the defect. Each step must name the specific UI element, table, column, or SQL query used. Implicit steps are not acceptable.

The Expected Behavior field is required and must state what the database or frontend should return upon completing the steps, based on the schema definition or the frontend component's data consumption logic.

The Actual Behavior field is required and must state exactly what the database or frontend returned instead, including any NULL values, empty arrays, error messages, or unexpected render outputs observed.

The Impact field is required for Critical and High severity defects and must explain the downstream consequence of the anomaly on the end user experience and on the Milestone 3 stability goals.

The Proposed Solution field is optional but strongly encouraged. The tester often has direct visibility into which nullable column or missing RLS policy caused the anomaly, and a proposed solution reduces the time the assigned developer spends diagnosing the root cause.

The Priority field is optional and is assigned by the Development Lead, not the Tester, to distinguish between defects of the same severity based on current sprint focus.

The Root Cause field is optional and is assigned by the Developer after investigation. Acceptable root cause categories for this project are: nullable column without frontend null guard, missing RLS policy, broken FK reference, incorrect JOIN type in frontend query, or stale seed data.
