
=== 4.2 BDD and BDD Tool Usage

===== Purpose

This section documents how Behavior-Driven Development (BDD) is applied in the *Tu Deporte Aquí* project to connect requirements (Section 2) with implementation and testing. It describes the artifacts that capture behavioral specifications, the current state of automated validation, and the trajectory for further integration.

===== BDD in the Project

BDD is a collaborative practice that uses concrete, plain‑language examples—written in Gherkin—to define expected system behavior before implementation. These examples serve simultaneously as specifications, acceptance criteria, and executable tests. In this project, BDD artifacts are organized into three layers:

* High‑level scenarios linked to functional and non‑functional requirements.
* Step definitions that bridge natural language to application code.
* A test runner (Cucumber) that executes scenarios and reports results.

===== Specification Artifacts

The repository contains three sets of `.feature` files, each serving a distinct purpose.

====== `features/` – Refined Requirements-Level Specifications

The directory `features/` at the repository root holds the most precise and up‑to‑date behavioral specifications. Three files cover the main requirement areas:

* `domain_requirements.feature` – domain rules for conflict detection, source authority, corrections, and data staleness.
* `interface_requirements.feature` – user‑visible indicators: timestamps, stale badges, missing‑data placeholders, source attribution, confidence displays.
* `machine_requirements.feature` – system behavior under load, failures, caching, logging, and recovery.

Every scenario in these files uses a consistent, domain‑specific vocabulary (e.g., “stale”, “conflicting reports”, “source down”, “partial data”) and is annotated with the requirement IDs it validates (e.g., `# Req: R-DOM-2, R-DOM-3`). These files constitute the canonical specification for the system’s behavior.

====== `tu-deporte-aqui/features/` – Intermediate Versions

The subdirectory `tu-deporte-aqui/features/` contains earlier versions of the same three feature files. The scenarios are substantially equivalent, but the terminology was less refined (e.g., “Continue operating when a source fails” rather than the current “Continue operating when a source is down”). These files were used to bootstrap the initial step‑definition stubs in `stepdefs.js`. They are retained for historical reference; the `features/` versions supersede them.

====== `tu-deporte-aqui/utils/.feature` – Utility‑Level Specification

The file `tu-deporte-aqui/utils/.feature` is an implementation‑oriented specification that directly reflects the existing score‑utility functions (`formatScore`, `mapGameStatus`, `isGameStale`). It contains numeric, concrete examples (e.g., “home score 80 and away score 75”, “timestamp older than threshold”) and tests the narrow slice of behavior that is already implemented. This file is not a replacement for the high‑level requirements scenarios; it documents the capabilities currently present in the utility layer.

===== Automation and Execution

Step definitions are maintained in `features/step_definitions/stepdefs.js`. Every Gherkin step from the requirements‑level feature files has a corresponding function. Steps for which the underlying logic has not yet been built return `'pending'`, so the Cucumber runner reports them explicitly as not yet implemented. Steps backed by existing code contain real assertions and are reported as passed or failed.

The Cucumber suite is executed with:

[source,bash]
----
npm run test-cucumber
----

At the time of writing, the full suite yields two passing scenarios and all others marked as pending.

===== Scenarios Validated as Passing

Only two high‑level scenarios currently pass, both from `features/interface_requirements.feature`. Their Gherkin is:

[source,gherkin]
----
Scenario: Indicate stale data to the user
  Given the game data is "35 minutes" old
  And the staleness threshold for live games is "30 minutes"
  When Sebastian views the game details page
  Then Sebastian should see a "Stale" badge next to the score
  And Sebastian should see an explanation about possible stale data
  And the stale indicator should use a muted color scheme

Scenario: Handle missing data gracefully
  Given no source has provided a score for this game
  When Sebastian views the game details page
  Then Sebastian should see a placeholder indicating score is not reported
  And Sebastian should not see an empty or "0-0" value
  And Sebastian should not see any error messages
----

These scenarios are fulfilled by the existing staleness‑detection and score‑formatting utilities. All remaining scenarios—including those for conflict resolution, source attribution, caching, metrics, and audit trails—remain in the pending state.

===== Current Implementation Coverage

The automated test results show that the majority of the behavioral specification is not yet implemented. Consequently, most functional and non‑functional requirements are unmet. This reflects the project’s scope relative to a single‑semester development cycle: the BDD infrastructure has been established, but the application logic needed to satisfy the full set of scenarios is incomplete.

===== Next Steps

Further adoption of BDD is planned by doing the following:

* *Scenario‑completion enforcement* – Institute a pull‑request policy requiring that every merged PR either adds a new Gherkin scenario or converts an existing pending scenario to passing. This ensures that code changes are always accompanied by corresponding behavioral specifications.
* *Automated reporting* – Develop a CI bot that comments on pull requests with the list of scenarios that changed status (pending → passing or passing → failing), making BDD progress visible to reviewers.
* *Step‑definition implementation* – Extend the step‑definition layer to cover the remaining scenarios as the underlying domain services (conflict detection, source reliability, caching, audit logs) are built.

===== Summary

The project uses BDD to maintain a direct link from requirements to executable specifications. The authoritative feature files reside in `features/`, while older versions and utility‑level specifications support the transition. Two interface‑related scenarios have been automated and pass; all others are pending.
