
=== 4.1 Test-Driven Development (TDD)

===== Purpose

This section documents a representative application of Test-Driven Development
(TDD) concepts within Tu Deporte Aquí.

Rather than claiming project-wide adoption of TDD, this section illustrates
how test-first thinking can support reliability, predictable behavior, and
incremental refinement.

The goal is to demonstrate the Red–Green–Refactor cycle using a limited and
representative example.

===== Representative TDD Workflow

TDD organizes development into three stages:

[cols="1,3"]
|===
|Stage |Description

|Red
|Define expected behavior through a failing test.

|Green
|Implement the minimum logic required to satisfy the test.

|Refactor
|Improve structure while preserving passing behavior.
|===

===== Example Application

To illustrate TDD principles, a representative formatting behavior was
selected.

Behavior under evaluation:

[source,javascript]
----
formatScore(homeScore, awayScore)
----

Expected outcomes:

[cols="2,2"]
|===
|Input |Expected Output

|(80,75)
|"80 - 75"

|(null,75)
|"Score unavailable"

|(80,null)
|"Score unavailable"

|(0,0)
|"0 - 0"
|===

===== Red Phase

Expected behavior is expressed before implementation.

[source,javascript]
----
expect(formatScore(null,75))
.toBe("Score unavailable");
----

The initial implementation does not satisfy all conditions.

===== Green Phase

Only the minimum logic required to satisfy the documented behavior is added.

[source,javascript]
----
function formatScore(homeScore, awayScore) {
    if (homeScore == null || awayScore == null)
        return "Score unavailable";

    return `${homeScore} - ${awayScore}`;
}
----

===== Refactor Phase

After behavior becomes stable, implementation may be simplified while
preserving successful outcomes.

Refactoring focuses on:

* reducing unnecessary conditions
* improving readability
* preserving existing behavior
* maintaining test consistency

===== Observed Limitations

This example illustrates selective application of TDD concepts.

Current limitations include:

* not all project functionality follows TDD
* examples remain representative rather than exhaustive
* testing coverage is still evolving

This section therefore does not claim complete adoption of TDD across the
project.

===== Relationship to Project Goals

Selective use of TDD concepts supports:

* clearer behavioral expectations
* lower ambiguity
* improved maintainability
* stronger alignment between expected and documented behavior

===== Summary

This section demonstrates how Test-Driven Development concepts can support
incremental and reliable development.

The example is intended to illustrate the workflow itself rather than to
represent complete project testing practices.
