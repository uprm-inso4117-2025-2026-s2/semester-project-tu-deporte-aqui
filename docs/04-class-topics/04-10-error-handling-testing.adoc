
=== 4.10 Error Handling Testing for Unreliable Sports Data

===== Purpose

This section defines expected system behavior when Tu Deporte Aquí encounters common data issues from unreliable sports sources: missing scores, delayed updates, conflicting information, and partial data. The focus is on defining testable system behavior, not implementation details.

===== Common Data-Related Errors

====== 1. Missing Scores

One or both teams' scores are unavailable from all sources due to timing, source unavailability, or data synchronization failures.

*Expected Behavior*:

* Scheduled games: Show game time. Message: "Scores available after game ends."
* Live games: Show last known score with timestamp. Mark as stale if > 10 minutes old.
* Finished games: Display game occurred but score unavailable. Link to external source.
* System never displays guess or default scores (e.g., "0-0").

====== 2. Delayed Updates

Sources publish updates significantly after game results become known, causing timing misalignment.

*Expected Behavior*:

* Fresh data (< 10 min old): Display normally.
* Moderately stale (10–30 min): Display with timestamp. Show "Refreshing..." indicator.
* Very stale (> 30 min): Mark as potentially outdated. Display timestamp prominently.
* Game finished: Accept update even if delayed, but show reported time.

====== 3. Conflicting Data Across Sources

Different sources report incompatible information for the same game due to update timing, manual entry errors, or tracking different subsets.

*Expected Behavior*:

* Minor difference (1–2 points): Accept most recent value. List all conflicting sources in detail view.
* Moderate difference (3+ points): Flag as conflict. Show both values with sources and timestamps.
* Major discrepancy (different winner): Isolate as critical error. Do not display either score. Alert admin.

====== 4. Incomplete or Partial Data

A source provides some fields but not others, or covers only a subset of teams/games.

*Expected Behavior*:

* Missing game time: Show score and teams. Display "Time unavailable."
* Missing standings teams: Show available teams with note: "[X] teams currently available."
* Missing metadata: Display core info (teams, score, status). Show "Additional details unavailable."

====== 5. Stale or Outdated Data

Data is correct but older than thresholds that make it unreliable for the game context.

*Expected Behavior*:

* Live game, < 5 min old: Display normally.
* Live game, 5–15 min old: Display with timestamp and "Refreshing..." indicator.
* Live game, > 15 min old: Apply warning icon and text. Suggest manual refresh.
* Finished games: Accept any age. No staleness warning.

====== 6. Data Corruption or Validation Failures

Data arrives in unexpected format, violates constraints, or contains impossible values (negative scores, invalid statuses, future dates).

*Expected Behavior*:

* Log error and do not display invalid data.
* Show user message in plain language: "Score data invalid. Check source."
* Continue functioning with remaining valid data sources.

===== Testing Strategies

====== Scenario-Based Testing (BDD)

Define Gherkin scenarios for each error type:

[source,gherkin]
----
Scenario: Missing score for live game
  Given a live game with no available scores
  When the game is displayed
  Then score field shows "Score unavailable"
  And game remains in schedule

Scenario: Conflicting scores from two sources
  Given Source A reports 80-75, Source B reports 79-75
  When the game is displayed
  Then conflict is flagged
  And both scores shown with source attribution
----

====== Boundary Testing

Test thresholds for staleness, conflict magnitude, and retry logic:

* Data 9 minutes old vs. 11 minutes old during live play
* Score difference 1 point vs. 10 points
* Missing 1 team vs. 5 teams in standings

====== State Transition Testing

* Scheduled → Live: Show last score or "Game starting"
* Live → Finished: Accept final score regardless of source timing
* Finished with conflict: Flag conflict but accept result
* Any state with stale data during live play: Mark with timestamp and warning

====== Source Reliability Testing

* Test all sources agreeing (baseline)
* Test one source differing
* Test majority vs. minority source disagreement
* Test all sources providing null (complete failure)

===== Application to Tu Deporte Aquí

Puerto Rico sports sources commonly fail in the following ways:

* Social media posts: delayed, manual entry errors, incomplete standings
* Semi-official websites: outdated 1–6 hours behind, inconsistent records
* Third-party aggregators: conflicting data, incomplete team coverage

Error handling priorities:

* Flag high-priority conflicts (score differences > 5 points)
* Mark data stale if > 10 minutes old during live play
* Display source attribution in all score displays
* Continue service in degraded mode if 1 of 3 sources fails
* Alert administrator if no sources have recent data

===== Implementation Guidance

All error handling must meet these requirements:

[options="header"]
|===
| Component | Requirement
| Input Validation | All source data passes validation before display
| Error Detection | System identifies missing, conflicting, and stale data
| User Messaging | Clear, domain-specific feedback for each error type
| Logging | All errors logged for administrative review
| Resilience | System continues in degraded mode if sources fail
|===

===== Conclusion

Error handling testing ensures Tu Deporte Aquí maintains data integrity and user trust when encountering unreliable sources. By identifying error scenarios, defining expected behavior, and mapping to testable acceptance criteria, the system becomes predictable, transparent, and reliable even under adverse data conditions.
