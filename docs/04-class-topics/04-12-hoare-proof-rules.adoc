=== 4.12 Applying Hoare Proof Rules to getGameLabelStatus

===== Purpose

This section applies Hoare proof rules from lecture to formally verify the correctness of the `getGameLabelStatus` function in `lib/label_status.ts`. The goal is to demonstrate that for any valid input, the function always produces the correct label as defined by the project's game status classification.

===== The Function

The `getGameLabelStatus` function takes a game status string and returns one of three labels that indicate the reliability of the data associated with that game:

- *Provisional*: the game is not yet finished (live, upcoming, or delayed)
- *Confirmed*: the game has finished (final)
- *Non oficial*: the status is unknown, unreliable, canceled, or unrecognized

The function first normalizes the input status using `normalizeGameStatus`, which maps raw status strings such as `"in_progress"`, `"scheduled"`, or `"postponed"` to a standard `NormalizedGameStatus` value. It then uses a switch statement to determine the appropriate label.

===== The Hoare Triple

A Hoare triple has the form `{ P } S { Q }`, where `P` is the pre-condition that must hold before the program `S` executes, and `Q` is the post-condition that must hold after execution.

For `getGameLabelStatus`, the triple is defined as follows:

*Pre-condition P*: `status` is any value of type `string`, `null`, or `undefined`.

*Program S*: `getGameLabelStatus(status)`

*Post-condition Q*:

- If `status` normalizes to `live`, `upcoming`, or `delayed`, then `result = "Provisional"`
- If `status` normalizes to `final`, then `result = "Confirmed"`
- If `status` normalizes to `canceled` or `unknown` (including null, undefined, or unrecognized strings), then `result = "Non oficial"`

===== Applying the IF Proof Rule

The IF proof rule states that to verify a conditional statement, each branch must be verified independently. For each branch, we assume the branch condition holds alongside the pre-condition, and verify that the post-condition is satisfied.

The switch statement in `getGameLabelStatus` has three branches. Each is verified below.

===== Branch 1: Provisional

*Branch condition*: the normalized status is `live`, `upcoming`, or `delayed`.

*Verification*: when the pre-condition holds and the normalized status falls into this group, the switch statement matches one of these cases and returns `"Provisional"`. The post-condition `result = "Provisional"` is satisfied. This branch holds.

===== Branch 2: Confirmed

*Branch condition*: the normalized status is `final`.

*Verification*: when the pre-condition holds and the normalized status is `final`, the switch statement matches this case and returns `"Confirmed"`. The post-condition `result = "Confirmed"` is satisfied. This branch holds.

===== Branch 3: Non oficial

*Branch condition*: the normalized status is `canceled`, `unknown`, or any unrecognized value (including null and undefined inputs, which `normalizeGameStatus` maps to `unknown`).

*Verification*: when the pre-condition holds and the normalized status falls into this group, the switch statement reaches the `canceled`, `unknown`, or default case and returns `"Non oficial"`. The post-condition `result = "Non oficial"` is satisfied. This branch holds.

===== Conclusion

All three branches of `getGameLabelStatus` satisfy the post-condition given the pre-condition. By the IF proof rule, the function is formally verified to be correct for any input of type `string`, `null`, or `undefined`. This confirms that the function reliably maps any game status value to the appropriate label as required by the project.
