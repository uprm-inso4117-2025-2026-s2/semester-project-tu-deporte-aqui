
==== 2.2.5 Machine Requirements

===== Purpose

This section defines the machine (non-functional) requirements for *Tu Deporte Aquí*. These requirements address performance, scalability, reliability, security, and operational aspects of the system. They are stated in measurable terms where possible; some thresholds are initial estimates and will be refined as the project progresses and more data becomes available.

===== Notes on Current Status

The requirements below represent the current understanding of system constraints and quality goals. As the project moves into implementation, some values (e.g., exact concurrency numbers, response time targets) may be adjusted based on load testing, user research, and infrastructure decisions.

===== Machine Requirements

[horizontal]
R-MAC-1 (Availability):: The system shall be available to users 24/7 with a target uptime of 99.5% (excluding planned maintenance). Planned maintenance windows shall be announced at least 24 hours in advance.

R-MAC-2 (Response Time - Normal Load):: Under normal load (concurrent users ≤ 100), the system shall respond to user-facing requests (e.g., loading game details, standings) within 200 milliseconds for 95% of requests.

R-MAC-3 (Response Time - Peak Load):: During peak events (e.g., playoffs, with concurrent users up to 1000), the system shall respond to critical requests (scores, game status) within 500 milliseconds for 90% of requests. Non-critical features may experience higher latency.

R-MAC-4 (Scalability):: The system shall be designed to handle an increase in concurrent users up to 2000 without significant degradation (response times may increase but remain below 2 seconds for critical requests). The architecture shall support horizontal scaling of stateless components.

R-MAC-5 (Data Freshness):: The system shall update live game scores from sources at least every 5 to 10 minutes during games, providing a safety margin before the 15-minute staleness threshold (R-DOM-7). For non-live data (standings, schedules), updates shall occur at least once per day. (Freshness thresholds for staleness marking are defined in R-DOM-7.)

R-MAC-7 (Error Logging and Retention):: The system shall log all errors occurring in the machine (e.g., source fetch failures, parsing errors, database connectivity issues) with machine-level detail: timestamp, source identifier, error type/code, and stack trace or diagnostic information. Error logs shall be persistently stored and retained for at least 90 days.

R-MAC-9 (Graceful Degradation):: If one or more data sources become unavailable, the system shall continue to serve cached data. The system shall flag cached data with staleness indicators (according to R-DOM-7) to communicate data age to dependent components.

R-MAC-10 (Data Retention):: The system shall retain canonical game data indefinitely for historical reference. Audit logs and conflict logs shall be retained according to policy (at least 30 days). Raw source snapshots may be retained for a shorter period (e.g., 7 days) for debugging.

R-MAC-11 (Security and Authentication):: All communication channels with external sources shall use HTTPS with certificate verification. Access to administrative functions shall require bearer tokens, API keys, or session tokens issued by an authentication service. Sensitive data (credentials, authentication tokens) shall be encrypted at rest using AES-256. Local authentication credentials, if stored, shall be hashed using bcrypt or equivalent with salt.

R-MAC-12 (Backup and Recovery):: The system's database shall be backed up daily, with backups retained for at least 30 days. Recovery procedures shall be documented and tested at least once per quarter.

R-MAC-13 (Testability):: The system shall be designed to support automated testing of reliability scenarios, including source failures, conflicting data, and stale data. Test harnesses shall allow simulation of these conditions.

R-MAC-14 (Progressive Web Application and Service Worker Support):: The system shall support installation as a progressive web application (PWA) and registration of service workers. Service workers shall receive and handle push notifications using the Push API with VAPID key authentication. On mobile platforms, PWA installation shall be required to enable push notification receipt; on desktop platforms, service workers may operate without PWA installation.

R-MAC-15 (Responsive Layout):: The application markup and stylesheets shall implement responsive design techniques to adapt the layout and display across different viewport widths. The application shall prioritize mobile viewport dimensions and maintain usability on screens with varying sizes from small mobile devices to desktop displays.

===== Open Items

The following aspects require further investigation:

* **Concurrency validation**: Precise concurrency numbers (R-MAC-2, R-MAC-3, R-MAC-4) should be validated through load testing against expected user base and traffic patterns. Current targets (100/1000/2000 concurrent users) are estimates pending infrastructure measurement.

* **Response time calibration**: Response time targets (R-MAC-2, R-MAC-3) may be adjusted based on actual network conditions, infrastructure capacity, and end-to-end latency measurements. No baseline metrics currently collected.

* **Log retention policy**: Log retention periods (R-MAC-7, R-MAC-10) require formalization of compliance requirements, regulatory obligations (if any), and audit trail lifecycle management.

* **Security and authorization expansion**: R-MAC-11 specifies authentication mechanisms (bearer tokens, HTTPS, encryption, hashing) and CSP headers are configured. However, authorization model (role-based access control, API key rotation, secrets management, cache poisoning hardening) requires detailed design and implementation.
