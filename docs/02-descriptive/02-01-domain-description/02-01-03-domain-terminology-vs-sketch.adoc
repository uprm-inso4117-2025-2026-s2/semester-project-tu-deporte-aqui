
==== 2.1.3 Domain Terminology in Relation to Domain Rough Sketch

===== Purpose

This section maps the observations in Section 2.1.1 to the standardized terms defined in Section 2.1.2. The goal is to keep one consistent vocabulary for domain description, requirements, and later analysis.

===== Mapping Method

Section 2.1.1 records raw situations in plain language. Section 2.1.2 defines precise terms for those same situations. This section connects both, so each rough-sketch concern is linked to specific terminology and no additional terms are introduced here.

===== Rough-Sketch Concern to Terminology Mapping

The mapping below shows how the broad concerns from Section 2.1.1 are expressed with the fixed terms from Section 2.1.2.

====== Concern A: Information arrives as scattered reports, not one clean feed

The rough sketch describes posts, screenshots, and updates coming from different origins and formats. In terminology, this is represented by:

* *Source*
* *Source Report*
* *Sporting Event*
* *Event Record*
* *Source Attribution*
* *Reliability Metadata*

This mapping keeps a clear distinction between what is reported (*Source Report*) and what the system currently maintains (*Event Record*).

====== Concern B: Missing, stale, or late information must not look confirmed

The rough sketch notes that users often cannot tell if data is missing, outdated, or pending confirmation. In terminology, this is represented by:

* *Verification State* (*Unverified*, *Confirmed*, *Final*)
* *Provisional*
* *Unknown*
* *Publish Time*
* *Ingest Time*
* *Last Updated Time*
* *Staleness*

This mapping ensures missing or old values are described explicitly instead of being presented with misleading certainty.

====== Concern C: Two or more reports can disagree

The rough sketch includes cases where the same event is shown differently across sources. In terminology, this is represented by:

* *Conflict*
* *Correction*
* *Verification State*
* *Confidence Level* (*High Confidence*, *Medium Confidence*, *Low Confidence*)
* *Silent Overwrite*

This mapping supports documenting disagreement, later correction, and traceable changes instead of silently replacing values.

====== Concern D: Dependability should be visible to users

The rough sketch emphasizes user trust and the need to interpret uncertain information quickly. In terminology, this is represented by:

* *Dependability Indicator*
* *Source Trust Level*
* *Confidence Level*
* *Source Attribution*

This mapping formalizes how the system communicates certainty, uncertainty, and origin to users.

====== Concern E: The system may continue operating under partial failure

The rough sketch includes delayed updates, incomplete coverage, and inconsistent availability. In terminology, this is represented by:

* *Degraded State*
* *Graceful Degradation*

This mapping states that the platform should continue providing qualified partial information instead of failing silently.

===== Consistency Rules for Later Sections

To keep terminology consistent across the document:

* Later sections should use the terms from Section 2.1.2 exactly as defined.
* Plain-language wording from Section 2.1.1 may be used for examples, but normative discussion should use standardized terms.
* If a new domain term is needed in future milestones, it must first be defined in Section 2.1.2 before being used elsewhere.
