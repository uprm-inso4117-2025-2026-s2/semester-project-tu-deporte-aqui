==== Function Signatures

These function signatures are tied to the action groups described in Section 2.1.5. They describe domain behavior, not database tables, API routes, UI widgets, or implementation classes.

Notation uses the form:

----
functionName : InputTypes → OutputTypes
----

The symbol `><` means that the function uses more than one domain value.

===== Domain Types Used in These Signatures

The signatures below use named domain types so that the reader can see what kind of information each function works with.

[cols="1,3", options="header"]
|===
| Type
| Meaning in this domain

| `SourceType`
| The kind of source that produced a report. Examples include league source, team page, reporter source, sports media source, third-party platform, or community source.

| `SourceReport`
| A specific reported value from one source, such as a score update, schedule change, standings update, or correction.

| `EventRecord`
| The current maintained representation of what is known about a sporting event.

| `SourceTrustLevel`
| A general assessment of how dependable a source tends to be.

| `PublishTime`
| The time when a source originally released the information.

| `IngestTime`
| The time when the information was recorded by the system.

| `LastUpdatedTime`
| The most recent time the event record changed.

| `VerificationState`
| The confirmation state of a reported value. Possible values include `Unverified`, `Confirmed`, and `Final`.

| `InformationStatus`
| The current condition of the information. Possible values include `Developing`, `Conflicting`, `Delayed`, `Corrected`, and `Unknown`.

| `ConfidenceLevel`
| The level of confidence assigned to a reported value. Possible values include `HighConfidence`, `MediumConfidence`, and `LowConfidence`.

| `ReliabilityMetadata`
| The domain information attached to a report or event record so that its source, timing, verification state, and confidence level can be understood together.

| `DependabilityIndicator`
| The visible dependability information that can be shown with a score, schedule, standings entry, or event update.

| `ConflictAssessment`
| The result of comparing reports that may disagree. It identifies whether the reports are consistent, conflicting, or still unresolved.

| `CorrectionTrace`
| A record of what changed when a correction replaces or updates previously reported information.

| `DegradedState`
| A system condition where the platform continues operating with missing, late, or incomplete source support.

| `GracefulDegradation`
| The behavior of continuing to show usable partial information while making the limitation visible.
|===

===== Typed Example: Assessing Dependability

The following function is a non-trivial example of using types to explain domain behavior:

----
assessReportDependability : SourceReport >< SourceTrustLevel >< EventRecord → VerificationState >< ConfidenceLevel >< ReliabilityMetadata
----

This signature shows that assessing a report is not a simple true/false decision. A report is judged using the report itself, the usual dependability of its source, and the current event record. The output separates `VerificationState` from `ConfidenceLevel`, so the documentation can describe cases such as an `Unverified` report with `LowConfidence` instead of hiding that distinction.

===== E1 Actions: Report Received

----
buildReliabilityMetadata : SourceType >< PublishTime >< IngestTime >< LastUpdatedTime → ReliabilityMetadata
----

Creates the reliability metadata attached to a newly received report.

----
assessReportDependability : SourceReport >< SourceTrustLevel >< EventRecord → VerificationState >< ConfidenceLevel >< ReliabilityMetadata
----

Assesses how dependable a received report is in relation to the current event record.

----
attachDependabilityIndicator : SourceReport >< ReliabilityMetadata → DependabilityIndicator
----

Creates the dependability information that can be shown with the reported value.

===== E2 Actions: Live Match State Updated

----
recordLiveStateUpdate : SourceReport >< EventRecord >< LastUpdatedTime → EventRecord >< ReliabilityMetadata
----

Updates the event record when a live-match report changes the current known state.

----
recordManualOverride : SourceReport >< Correction >< EventRecord >< ReliabilityMetadata → Corrected >< EventRecord >< CorrectionTrace >< ReliabilityMetadata
----

Records a manual override as a correction when local updates are irregular, late, or inconsistent.

===== E3 Actions: Data Missing or Partial Coverage Detected

----
classifyPartialCoverage : EventRecord >< LastUpdatedTime >< ReliabilityMetadata → InformationStatus
----

Classifies incomplete coverage as `Developing`, `Delayed`, or `Unknown` depending on timing and available source context.

----
preserveLastKnownMetadata : EventRecord >< ReliabilityMetadata → EventRecord >< ReliabilityMetadata
----

Keeps the last known reliability context attached to the event record when current coverage is partial.

===== E4 Actions: Conflict Detected

----
detectConflict : SourceReport >< SourceReport >< EventRecord → ConflictAssessment
----

Compares two reports about the same event record and determines whether their values can both be accepted.

----
lowerConfidenceAfterConflict : ConflictAssessment >< ReliabilityMetadata → LowConfidence >< ReliabilityMetadata
----

Lowers confidence when the conflict is still unresolved.

----
revalidateConflictingInformation : ConflictAssessment >< SourceReport >< SourceTrustLevel → VerificationState >< ConfidenceLevel >< ReliabilityMetadata
----

Re-evaluates conflicting information when another report or more dependable source becomes available.

===== E5 Actions: Data Becomes Stale

----
markAsDelayed : EventRecord >< LastUpdatedTime >< ReliabilityMetadata → Delayed >< ReliabilityMetadata
----

Marks information as delayed when it has not been updated recently enough for its context.

----
preserveLastKnownReliabilityMetadata : Delayed >< ReliabilityMetadata → DependabilityIndicator
----

Keeps the previous reliability context visible instead of removing stale information without explanation.

===== E6 Actions: Correction Published

----
recordCorrection : Correction >< EventRecord >< ReliabilityMetadata >< LastUpdatedTime → Corrected >< EventRecord >< CorrectionTrace >< ReliabilityMetadata
----

Records that previously published information changed and keeps a trace of the correction.

----
explainCorrection : CorrectionTrace >< ReliabilityMetadata → DependabilityIndicator
----

Connects the correction trace to the dependability information shown with the corrected value.

===== E7 Actions: Source Outage / Unavailable Source

----
enterDegradedState : EventRecord >< ReliabilityMetadata → DegradedState
----

Records that source support is not enough for normal operation.

----
maintainGracefulDegradation : DegradedState >< EventRecord >< ReliabilityMetadata → GracefulDegradation >< DependabilityIndicator
----

Keeps partial information available while showing that the system is operating under degraded conditions.

----
avoidSilentFailure : DegradedState >< EventRecord → GracefulDegradation
----

Prevents missing or delayed information from being hidden as if nothing were wrong.

===== Note on Scope

These signatures stay at the domain level. The types name concepts from the problem area: reports, sources, event records, verification, confidence, corrections, conflicts, and degraded operation. They do not define code classes, database records, storage queues, or UI components.
