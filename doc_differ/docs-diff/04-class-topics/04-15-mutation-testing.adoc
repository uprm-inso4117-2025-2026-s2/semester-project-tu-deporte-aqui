
=== 4.15 Mutation Testing

==== Purpose

[.inserted]#Mutation testing evaluates the effectiveness of existing automated tests by introducing small intentional modifications (mutations) into the source code and verifying whether the implemented tests or observable system behavior detect the modified logic.#

[.inserted]#This technique helps identify weak or incomplete testing scenarios that may still pass despite incorrect implementation behavior.#

==== Relevance to the Project

[.inserted]#In the project, mutation testing is relevant because the platform depends on reliable handling of sports data, status ordering, and source-aware information.#

[.inserted]#Small unnoticed logic changes could produce:#


* [.inserted]#incorrect standings#
* [.inserted]#invalid score ordering#
* [.inserted]#inconsistent UI behavior#
* [.inserted]#incorrect API responses#
* [.inserted]#stale or conflicting information display#

[.inserted]#Mutation testing helps validate whether the implemented testing and validation strategies are capable of detecting these behavioral regressions.#

==== Mutation Testing Approach

[.inserted]#The mutation testing analysis focused on the `/api/games` route responsible for retrieving and processing sports data.#

[.inserted]#Several intentional code mutations were introduced to simulate realistic implementation faults.#

[.inserted]#The evaluation observed:#


* [.inserted]#frontend behavior#
* [.inserted]#API responses#
* [.inserted]#terminal/server logs#
* [.inserted]#visible UI inconsistencies#

[.inserted]#A mutation was considered:#


* [.inserted]#*Killed* if the fault produced observable incorrect behavior#
* [.inserted]#*Survived* if the system behaved as if nothing was wrong#
