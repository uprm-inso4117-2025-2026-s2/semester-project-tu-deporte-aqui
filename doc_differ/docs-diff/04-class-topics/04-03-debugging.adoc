
=== 4.3 Use of Debugging

===== Purpose

This section outlines the expected debugging practices that will support reliability and transparency in Tu Deporte Aquí. Debugging will help verify that the system correctly handles conflicting data, stale updates, and source failures—key reliability concerns identified in our prior A3 submission. These scenarios represent the types of debugging sessions we expect to conduct during implementation.

===== Role of Debugging

Debugging provides visibility into data flows, especially when sources disagree or updates arrive late. It allows us to examine failure paths before they reach users.

Key contributions:

- Verify conflict detection logic: ensures inconsistent scores don't confuse fans.
- Confirm staleness thresholds trigger transparency indicators: shows users how fresh the data is.
- Inspect state during partial source failures: helps maintain service even if some sources fail.
- Validate degraded mode messaging: informs users when data is limited or unavailable.

===== Debugging Environment

All scenarios use:

- **Browser DevTools** for frontend inspection and network analysis
- **IDE debugger** for backend code examination
- **Debug console** for expression evaluation during breakpoints

===== Expected Debugging Scenarios

The following scenarios represent examples of debugging sessions we plan to conduct during implementation to validate core system behaviors. Each scenario identifies a reliability concern, describes the hypothesis we will test, and outlines which debugging techniques will help us verify the behavior.

===== Scenario 1: Score Display with Missing Data

*Hypothesis*: When one of the scores is missing, the system should display "Score unavailable" instead of an incorrect score, making sure fans are not mislead by incomplete data.

*Debugging Session*:

- A breakpoint was set inside `buildDisplayGame` before the return statement.
- A test case was executed with `homeScore = null` and `awayScore = 75`.
- The debugger was stepped into `isScoreComplete` to verify how completeness was evaluated.
- The **watch panel** was used to inspect `homeScore`, `awayScore`, and `scoreComplete`.
- Execution continued into `formatScore` to confirm whether it was called.

*Observed Outcome*:

- It was observed that `isScoreComplete` returned `false` when `homeScore` was `null`.
- As a result, `formatScore` was not executed.
- The value `"Score unavailable"` was correctly assigned to `displayScore`.

===== Scenario 2: Staleness Threshold Behavior

*Hypothesis*: Data older than 10 minutes during live play should be marked `Stale` and shown in UI. fans want fresh data, older information should be marked stale.

*Debugging Session*:

- A breakpoint was set inside `isStaleTimestamp`.
- A test case was executed with a timestamp older than 20 minutes.
- The **watch panel** was used to monitor `lastUpdatedDate`, `now`, and `diffMinutes`.
- Step through to verify that `diffMinutes` exceeds `thresholdMinutes`.
- The **debug console** was used to evaluate `now - lastUpdatedDate` in milliseconds.

*Observed Outcome*:

- The calculated `diffMinutes` exceeded the threshold value.
- The function returned `true`, indicating stale data.

===== Scenario 3: Full Workflow Integration

*Hypothesis*: `buildDisplayGame` correctly integrates all utility functions to produce a consistent UI-ready object.

*Debugging Session*:

- A breakpoint was set at the beginning of `buildDisplayGame`.
- A complete game object was passed with valid scores, status, and timestamp.
- The debugger stepped into each helper function: `mapGameStatus`, `isScoreComplete`, and `isStaleTimestamp`.
- The **call stack** was inspected to trace execution flow across functions.
- The **watch panel** was used to monitor intermediate values.

*Observed Outcome*:

- Each helper function returned expected values.
- `scoreComplete` was `true`, and `formatScore` produced the correct formatted score.
- The final object contained consistent and expected values for all fields.

===== Integration with Testing

Debugging will complement automated tests during implementation:

- When reliability tests fail, we will debug to inspect intermediate state.
- We will explore edge cases difficult to capture in tests.
- We will validate behavior before writing acceptance tests.

===== Alignment with Project Goals

Defining debugging practices early establishes clear expectations for behavior examination during development. These scenarios directly support the reliability and transparency focus of our A3, enabling us to systematically verify that fault-injection and stress-testing practices are working as intended.

===== Conclusion

Debugging will be a critical tool for ensuring Tu Deporte Aquí behaves predictably under uncertainty. By examining conflict detection, staleness, and degraded mode during implementation, we will build confidence that reliability requirements are verifiable and reducible to testable code behavior.
