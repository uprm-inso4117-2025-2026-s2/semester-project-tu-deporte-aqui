== [.inserted]#1. Introduction#

[.inserted]#This lecture topic task evaluates how sensitive the current system is to faults by applying mutation testing to a core API route.#

[.inserted]#Instead of only checking whether the system works under normal conditions, mutation testing introduces small, controlled faults into the code. The goal is to observe whether these faults are detectable through the current behavior of the system.#

[.inserted]#The focus of this experiment is the `/api/games` route, which retrieves sports data from Supabase and feeds the frontend.#

== [.inserted]#2. Scope of the Experiment#

[.inserted]#The experiment was conducted on:#

- [.inserted]#API route: `/api/games`#
- [.inserted]#Data source: Supabase (games table)#
- [.inserted]#Frontend page: `/` (games display UI)#

[.inserted]#The evaluation considered three observable outputs:#

- [.inserted]#API response (`/api/games`)#
- [.inserted]#Frontend behavior (`/`)#
- [.inserted]#Server logs (terminal output)#

[.inserted]#No automated testing framework was used. Detection was based on observable system behavior.#

== [.inserted]#3. Mutation Testing Approach#

[.inserted]#Each mutation consisted of modifying a small part of the code to simulate a realistic fault. After each change:#

[.inserted]#1. The application was executed (`npm run dev`)#
[.inserted]#2. The API endpoint was tested (`/api/games`)#
[.inserted]#3. The frontend was observed (`/`)#
[.inserted]#4. Server logs were checked#

[.inserted]#A mutation is considered:#

- [.inserted]#*Killed* → if the fault causes a visible or detectable change#
- [.inserted]#*Survived* → if the system behaves as if nothing is wrong#

== [.inserted]#4. Mutations Applied#

== [.inserted]#5. Detailed Observations#

=== [.inserted]#M1 – Invalid Table Name#

[.inserted]#Changing the table name caused the query to fail immediately.#

- [.inserted]#API response:#
[.inserted]#  {"error":"Could not find the table 'public.game'..."}#

- [.inserted]#Frontend:#
[.inserted]#  Error al cargar los juegos#

- [.inserted]#Terminal:#
[.inserted]#  GET /api/games 500#

[.inserted]#This confirms that database-level failures are clearly detectable.#

---

=== [.inserted]#M2 – Inverted Error Condition#

[.inserted]#The system treated successful queries as failures.#

- [.inserted]#API response:#
[.inserted]#  {"error":"Server error"}#

- [.inserted]#Frontend:#
[.inserted]#  Error al cargar los juegos#

[.inserted]#This shows that control-flow errors in error handling are visible at runtime.#

---

=== [.inserted]#M3 – Incorrect Status Code#

[.inserted]#The API returned valid data but used HTTP 500.#

- [.inserted]#`/api/games` displayed data#
- [.inserted]#`curl -i` confirmed HTTP 500#
- [.inserted]#Frontend still failed#

[.inserted]#This demonstrates that the system depends on correct HTTP status codes, not only response content.#

---

=== [.inserted]#M4 – Missing Required Field#

[.inserted]#Removing `status` did not crash the system.#

- [.inserted]#API returned valid data without `status`#
- [.inserted]#Frontend rendered but lost status values#

[.inserted]#This is a non-crashing fault that still affects correctness.#

---

=== [.inserted]#M5 – Modified Error Structure#

[.inserted]#The error response format changed.#

- [.inserted]#API response:#
[.inserted]#  {"message":"Could not find the table..."}#

- [.inserted]#Frontend still failed#
- [.inserted]#HTTP 500 still returned#

[.inserted]#This confirms that the system detects failures even if the error format changes.#
