= [.inserted]#Data Staleness Detection Strategies#
[.inserted]#By Noel Colon#

== [.inserted]#1. Introduction#

[.inserted]#Tu Deporte Aquí combines sports information from multiple leagues, reporters, and informal sources across Puerto Rico.#

[.inserted]#Because the platform relies on continuously updated external data, it is vulnerable to situations where information stops updating without clear signals of failure.#

[.inserted]#The purpose of this research is to:#

* [.inserted]#Identify stale sports data#
* [.inserted]#Establish freshness expectations#
* [.inserted]#Recommend system behaviors that preserve transparency and reliability when freshness violations occur#

== [.inserted]#2. Problem Context#

[.inserted]#Stale data can emerge in many scenarios, including:#

* [.inserted]#A game remains marked as "Live" after completion#
* [.inserted]#Standings fail to reflect completed matches#
* [.inserted]#Player statistics are not updated following a game#
* [.inserted]#A data source silently stops publishing updates#

== [.inserted]#3. Detection Strategies#

[cols="1,1,2", options="header"]
[.inserted]#|===#
[.inserted]#|Data Type |Expected Update |Staleness Threshold#

[.inserted]#|Live Score#
[.inserted]#|Frequent#
[.inserted]#|10–15 minutes without change#

[.inserted]#|Game State#
[.inserted]#|Event-driven states#
[.inserted]#|30 minutes past scheduled start OR 3 hours after start if still marked live#

[.inserted]#|Standings#
[.inserted]#|Post match#
[.inserted]#|24–48 hours#

[.inserted]#|Player Statistics#
[.inserted]#|Post match#
[.inserted]#|24 hours#
[.inserted]#|===#

=== [.inserted]#3.1 Time-Based Check#

[.inserted]#Flags data as stale when sufficient time has passed since the last update.#

==== [.inserted]#Pros#

* [.inserted]#Easy to implement#
* [.inserted]#Predictable behavior#

==== [.inserted]#Cons#

* [.inserted]#May produce false positives#
* [.inserted]#Does not catch stale states unless time threshold is exceeded#

=== [.inserted]#3.2 Lifecycle Check#

[.inserted]#Marks data as stale when an expected state transition does not occur.#

==== [.inserted]#Pros#

* [.inserted]#More context-aware#
* [.inserted]#Effective for game state and live score#

==== [.inserted]#Cons#

* [.inserted]#Requires defining expected duration of games/states#
* [.inserted]#Different sports have different durations#
* [.inserted]#Harder to implement#

=== [.inserted]#3.3 Source Activity Check#

[.inserted]#Flags staleness depending on whether the data source is still publishing updates.#

==== [.inserted]#Pros#

* [.inserted]#Can detect multiple stale data points at once#
* [.inserted]#Useful for evaluating source reliability#

==== [.inserted]#Cons#

* [.inserted]#Source may still be active but not updating specific events#
* [.inserted]#Requires continuous monitoring of source activity#

== [.inserted]#4. Recommended Strategy#

[.inserted]#The strongest approach is a *hybrid detection strategy* combining:#

* [.inserted]#Time-Based Check#
* [.inserted]#Lifecycle Check#
* [.inserted]#Source Activity Check#

[.inserted]#Benefits:#

* [.inserted]#Covers more edge cases#
* [.inserted]#Reduces blind spots#
* [.inserted]#Improves overall reliability and transparency#

[.inserted]#Trade-off:#

* [.inserted]#More complex implementation#
