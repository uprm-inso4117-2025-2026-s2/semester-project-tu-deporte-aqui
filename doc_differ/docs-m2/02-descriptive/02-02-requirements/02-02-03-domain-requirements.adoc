
==== 2.2.3 Domain Requirements

===== Purpose

This section defines the domain requirements for *Tu Deporte Aquí*. These requirements capture the essential phenomena and concepts that the system must faithfully simulate, grounded in observations from the domain description (section 2.1) and the epics and user stories (section 1.2).

===== Domain Requirements

[horizontal]
R-DOM-1 (Source Authority - Automated Sources):: External data sources shall be assigned trust levels based on type: Official sources (league websites, official announcements) shall have high trust; News outlets and aggregators shall have medium trust; Social media and user-generated sources shall have low trust.

R-DOM-1a (Source Authority - Manual Input):: Data manually entered by authorized users shall be assigned trust levels based on role: League representatives and system administrators shall have high trust; Reporters and coaches/organizers shall have medium trust; Authorized community members shall have low trust. Manual entries shall be properly attributed and timestamped by role.

R-DOM-2 (Conflict Detection):: The system shall detect when two or more sources report different values for the same game attribute (e.g., score, status) and mark that attribute as conflicting. Multiple sources reporting the same value (even with different timestamps) do not constitute a conflict.

R-DOM-3 (Conflict Preservation and Resolution):: When a conflict is detected, the system shall preserve all reported values and their source attributions. Three timestamp types shall be tracked: Event Time (when the event occurred in the domain), Ingestion Time (when the data entered the system), and Reported Time (when the source reported it). All reported values shall be visible together, with the highest-trust source and most recent reported time emphasized. The canonical value shall not be automatically overwritten; all conflicting values are retained for manual or rule-based resolution.

R-DOM-4 (Unverified Data State):: Data from non-high-authority sources shall be placed in the unverified verification state, indicating it has not been confirmed by an official source. Unverified data shall be distinguished from confirmed data (See R-INT-9). When an official source confirms such data, the verification state shall transition from unverified to confirmed.

R-DOM-5 (Confidence Calculation):: Confidence levels for data shall be calculated from verification state, source type authority, and staleness according to the following rules: If an official source exists and there are no conflicts and data is not stale, confidence shall be High. If multiple unofficial sources agree and there are no conflicts and data is not stale, confidence shall be Medium. If data comes from a single unverified source, or if there are unresolved conflicts, or if all sources are inactive, or if data is stale, confidence shall be Low. Otherwise, confidence defaults to Low.

R-DOM-6 (Corrections and Audit Trail):: When an official source issues a correction to a previously published value, the system shall update the canonical value and create an audit trail entry with the previous value, reason (e.g., "official correction"), source, and timestamp.

R-DOM-7 (Staleness):: The system shall track the last update time for each game attribute and mark data as stale according to the following thresholds: For live game scores, if no update is received within 15 minutes, the data shall be marked as stale. For game state transitions (R-DOM-9), if no state change is received within 30 minutes past the scheduled game start time, or within 3 hours after the game is marked Live, the state shall be marked as stale. For standings, if no update is received within 24 hours, the data shall be marked as stale. For schedules, if no update is received within 48 hours, the data shall be marked as stale. When data transitions to stale state, it shall trigger confidence recalculation (R-DOM-5) and the stale status shall be communicated to dependent components via staleness indicators (see R-INT-9).

R-DOM-8 (Source Unavailability):: If a source ceases to provide updates (e.g., repeated fetch failures), the system shall mark that source as inactive. The confidence level of data shall reflect source availability via the staleness and confidence mechanisms defined in R-DOM-5 and R-DOM-7.

R-DOM-9 (Game State Transitions):: The system shall model game states (scheduled, live, halftime, overtime, delayed, suspended, final, postponed, cancelled) and enforce valid state transitions. Valid transitions are: Scheduled → Live (at start time); Live → Halftime (at regulation end); Halftime → Live (when break ends); Live → Overtime (if tied at regulation end); Live/Overtime → Final (at game completion); Live → Delayed (for technical interruptions); Live → Suspended (for indefinite pause); Suspended → Live (when resumed); Scheduled → Postponed (before starting); Scheduled → Cancelled (before starting); any state → Final (for corrections). Each state transition shall be recorded with timestamp, source, and reason.

R-DOM-10 (Audit Trail for Critical Data):: All changes to critical game data (score, status, final result) must be recorded in an audit trail. Each entry shall include: previous value, new value, source of the change (automated source or user), timestamp, and reason (e.g., "official correction", "user update").

R-DOM-11 (Conflict Logging):: When the system detects conflicting reports from multiple sources, a conflict log entry shall be created with: both conflicting values, the sources reporting each, the timestamps of each report (event time), and whether the conflict has been manually resolved. Conflict logs shall be retained for audit and analysis purposes.

R-DOM-12 (User Attribution):: All manual changes to domain data shall be attributed to the authenticated user who performed the change. The audit trail must record the user ID (or role if applicable) along with each manual change.
